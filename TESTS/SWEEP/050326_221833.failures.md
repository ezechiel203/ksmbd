## CATEGORY D: Server Bugs — ACL / Security

### smb2.acls.CREATOR
**Error:** `acls.c:167: NT_STATUS_OK should be NT_STATUS_ACCESS_DENIED`
**Cause:** CREATOR OWNER ACE is not enforced. When a CREATOR OWNER ACE denies access, ksmbd still returns OK.
**Fix:** In `smbacl.c`, implement CREATOR OWNER/GROUP ACE expansion: replace the CREATOR OWNER SID with the actual owner SID when evaluating inherited ACEs.

### smb2.acls.OWNER
**Error:** `acls.c:774: NT_STATUS_OK should be NT_STATUS_ACCESS_DENIED`
**Cause:** OWNER RIGHTS ACE (`S-1-3-4`) not evaluated. The access check ignores OWNER RIGHTS ACEs.
**Fix:** In `smbacl.c` ACE iteration, handle the OWNER RIGHTS SID (`SE_OWNER_RIGHTS`): when present, it replaces implicit owner access; deny bits must be respected.

### smb2.acls.ACCESSBASED
**Error:** `acls.c:2188: Unable to connect to share 'hideunread'`
**Cause:** The `hideunread` share doesn't exist in `ksmbd.conf` on the VM.
**Fix:** Add to `ksmbd.conf`:
```
[hideunread]
  path = /srv/smb/hideunread
  read only = no
  access based share enum = yes
```

### smb2.acls_non_canonical.flags
**Error:** `Missing SEC_DESC_DACL_AUTO_INHERITED (0x0400) in sd_orig->type`
**Cause:** When inheriting ACEs from a parent, ksmbd doesn't set the `SE_DESC_DACL_AUTO_INHERITED` bit in the DACL type field.
**Fix:** In `smbacl.c`, after propagating inherited ACEs, set `SEC_DESC_DACL_AUTO_INHERITED` in `sd->type`.

### smb2.maximum_allowed.maximum_allowed
**Error:** `max_allowed.c:162: NT_STATUS_ACCESS_DENIED should be NT_STATUS_OK`
**Cause:** `MAXIMUM_ALLOWED` access mask is not being resolved to the caller's actual allowed permissions. ksmbd may be passing it as-is to the VFS instead of computing the actual mask.
**Fix:** In `smb2_create.c` access resolution, detect `MAXIMUM_ALLOWED` in DesiredAccess, compute the allowed access mask from the DACL (what the user can actually get), then open with that mask.

### smb2.secleak.secleak
**Error:** `secleak.c:85: Invalid credentials should have failed`
**Cause:** Authentication with deliberately invalid/corrupt NTLMSSP credentials is succeeding. May be a regression from the anonymous auth fix (zero-length NtChallengeResponse now accepted).
**Fix:** In `auth.c`, ensure that non-anonymous NTLMSSP authentication with an invalid NT hash is properly rejected. The anonymous path (zero-length response) should only be taken when the username is empty/guest.

### smb2.session-require-signing.bug15397
**Error:** `security_mode was 1 (SIGNING_ENABLED), expected 3 (SIGNING_REQUIRED)`
**Cause:** When the client sets `SMB2_NEGOTIATE_SIGNING_REQUIRED` in its SecurityMode, the server must also set `SIGNING_REQUIRED` in the negotiate response. ksmbd only sets `SIGNING_ENABLED`.
**Fix:** In `smb2_negotiate.c`, if client SecurityMode has `SMB2_NEGOTIATE_SIGNING_REQUIRED` set, respond with `SMB2_NEGOTIATE_SIGNING_REQUIRED | SMB2_NEGOTIATE_SIGNING_ENABLED`.

## CATEGORY F: Server Bugs — Leases & Oplocks

### smb2.lease.statopen / statopen4
**Error:** `got lease_state 0x0 should be 0x7 (RWH)`
**Cause:** Stat opens (DesiredAccess = READ_CONTROL or FILE_READ_ATTRIBUTES only) are not being granted leases. ksmbd's oplock path checks for write access before granting W caching — but for stat opens, even RH should be granted.
**Fix:** In `smb_grant_oplock()` (`oplock.c`), do not deny lease grant based on DesiredAccess being read-only. The lease state should be determined by the sharing mode negotiation, not by the open's access mask.

### smb2.lease.upgrade / upgrade2 / upgrade3
**Error:** `got 0x3 should be 0x7`, `got 0x5 should be 0x7`, `got 0x1 should be 0x3`
**Cause:** Lease upgrade from a lower state (R→RWH, RH→RWH, etc.) is not working. When a second CREATE for the same lease key requests a higher state, ksmbd should upgrade the existing lease.
**Fix:** In `smb_grant_oplock()`, when an existing lease is found for the same lease key and the new request asks for a superset of the current state, upgrade the lease state atomically.

### smb2.lease.oplock
**Error:** `got 0x3 (RH) should be 0x1 (R)` after oplock conflict
**Cause:** When an oplock conflicts with a lease, the lease should be downgraded to R-only (no handle caching). ksmbd preserves the H bit.
**Fix:** In the oplock conflict resolution path, strip `SMB2_LEASE_HANDLE_CACHING_LE` from the conflicting lease when an oplock is granted to a competing opener.

### smb2.lease.breaking2 / breaking3 / v2_breaking3 / breaking4
**Error:** `NT_STATUS_INVALID_NETWORK_RESPONSE` or `req2->state != SMB2_REQUEST_RECV`
**Cause:** Lease break acknowledgment flow is broken. `breaking2` gets INVALID_NETWORK_RESPONSE (wrong break opcode format). `breaking3/v2_breaking3` have a pending request stuck in RECV state after the break ack. `breaking4` has a request that should be done but isn't.
**Fix:**
- `breaking2`: The LEASE_BREAK_ACK response format is incorrect — check the response structure in `smb2_misc_cmds.c` lease break ack handler.
- `breaking3/v2_breaking3/breaking4`: After the break ack is processed, wake all threads blocked on `opinfo->pending_break`. Ensure `smb2_oplock_break_noti()` / `smb2_lease_break_noti()` properly signals the blocked opener.

### smb2.oplock.batch3 / batch7
**Error:** `NT_STATUS_SHARING_VIOLATION, expected NT_STATUS_OK`
**Cause:** When a second client opens a file that has a batch oplock, the server should break the oplock and hold the second open until the break is acknowledged (or timeout). Instead ksmbd immediately returns SHARING_VIOLATION.
**Fix:** In `smb_grant_oplock()`, when a batch oplock conflict is detected, send the break notification and defer (`wait_for_break_ack()`) instead of returning -EACCES.

### smb2.kernel-oplocks.kernel_oplocks2 / kernel_oplocks4 / kernel_oplocks5 / kernel_oplocks7
**Error:** `NT_STATUS_SHARING_VIOLATION or wrong oplock level` when kernel has local file open
**Cause:** ksmbd doesn't integrate with the Linux kernel's `F_SETLEASE`/`F_RDLCK` mechanism. When a local kernel file is open with a lease, SMB opens should break the local lease first.
**Fix:** In `ksmbd_vfs_open_file()`, use `fcntl(F_SETLEASE, F_UNLCK)` to break any existing kernel lease before granting an SMB oplock. This requires the ksmbd fd to participate in the kernel's lease protocol (`lease_notifier` callback). Complex change — requires kernel VFS cooperation.

## CATEGORY G: Server Bugs — Compound & Async

### smb2.compound_async.write_write / read_read
**Error:** `req[1]->cancel.can_cancel=0 should be 1`
**Cause:** The second operation in a compound write/write or read/read is being processed synchronously (not async). For it to be cancellable, it must be dispatched as an async request with `SMB2_HDR_FLAG_ASYNC`.
**Fix:** In `smb2_write()`/`smb2_read()` for compound requests, when the operation would block (e.g., lock contention), ensure it is registered as an async request so `can_cancel` is set.

### smb2.compound_async.getinfo_middle
**Error:** `NT_STATUS_FILE_CLOSED, expected NT_STATUS_OK`
**Cause:** In a compound where GETINFO is in the middle of the chain using the compound FID from a preceding CREATE, the FID is not being propagated to GETINFO. The FID propagation for GETINFO uses a different code path than SET_INFO.
**Fix:** In `init_chained_smb2_rsp()` (`smb2_pdu_common.c`), ensure that when `compound_fid` is set from a CREATE, it is forwarded to GETINFO (`SMB2_GET_INFO`/`SMB2_QUERY_INFO`) as well.

### smb2.compound_async.rename_non_compound_no_async / rename_last / rename_middle
**Error:** `smb2_lease_break_ack failed` or `req state not SMB2_REQUEST_RECV`
**Cause:** Rename operations in an async compound trigger a lease break. The break ack fails or the pending request state is wrong. The lease break path and the compound async response path are interfering.
**Fix:** Ensure that when a rename triggers a lease break inside a compound request, the break notification is sent and the compound response is only generated after the break is processed (or the break timer expires).

## CATEGORY L: Server Bugs — DOS Attributes & Timestamps

### smb2.dosmode.dosmode
**Error:** `FILE_ATTRIBUTE_HIDDEN is not set after SET_INFO`
**Cause:** `FILE_ATTRIBUTE_HIDDEN` is set via SMB2_SET_INFO but not persisted. The xattr write may be skipped for HIDDEN or the xattr is not read back correctly.
**Fix:** In `smb2_query_set.c` FileBasicInformation handler, ensure `FILE_ATTRIBUTE_HIDDEN` is included in the xattr write. Check `ksmbd_vfs_setxattr()` call for `XATTR_NAME_DOS_ATTRIBUTE`.

### smb2.openattr.openattr
**Error:** `got attr 0x20 (ARCHIVE), should be 0x21 (ARCHIVE|READONLY)` after truncate
**Cause:** On file truncate (SetEndOfFile to 0), ksmbd clears `FILE_ATTRIBUTE_READONLY`.
**Fix:** In the `smb2_set_info.c` FileEndOfFileInformation path, do not modify the existing attribute flags. Only update size.

### smb2.timestamp_resolution.resolution1
**Error:** `write_time precision error — wrong rounding`
**Cause:** The timestamp returned in the file info doesn't match expected resolution. ksmbd may be returning nanosecond-precision timestamps from Linux `timespec64` without rounding to 100ns Windows FILETIME resolution.
**Fix:** Ensure all timestamps returned in `smb2_query_set.c` / `ksmbd_info.c` are rounded to 100ns intervals (divide by 100, multiply by 100 in the `unix_to_NT_time()` conversion).

### smb2.timestamps.freeze-thaw
**Error:** `create_time was 0xFFFFFFFFFFFFFFFF expected actual time`
**Cause:** After a freeze/thaw operation (setting timestamps to -1 means "don't change"), `create_time = 0xFFFFFFFFFFFFFFFF` is being stored literally instead of being treated as "no change".
**Fix:** In `FileBasicInformation` SET_INFO handler, skip updating `create_time` when it is `0xFFFFFFFFFFFFFFFF` (MS-FSA: means "do not change"). Same check already exists for other timestamps — apply to create_time too.

### smb2.timestamps.modern_write_time_update-1 / -2
**Error:** `currenttime1 is wrong`
**Cause:** The deferred write time update (MS-SMB2 §3.3.5.20.3 — write time updated 2s after last write) is not implemented or the timer fires at wrong time.
**Fix:** Implement deferred write time: on SMB2 WRITE, don't update `mtime` immediately; schedule a 2-second timer that updates `mtime` to the time of the last write. This is the "lazy write time" semantics Windows uses.

### smb2.charset.* (4 failures: composite, diacritical, surrogate, wide-a)
**Error:** `NT_STATUS_OBJECT_NAME_INVALID for Unicode composed characters`
**Cause:** ksmbd's filename validation rejects UTF-16LE composed Unicode characters (precomposed NFC, combining diacriticals, surrogate pairs, fullwidth forms). Linux filesystems accept arbitrary byte sequences in filenames.
**Fix:** In the filename validation path (`smb2_create.c` / `ksmbd_validate_filename()`), relax the Unicode character validation. Do not reject code points based on Unicode category — only reject actual null bytes and path separators (`/`, `\0`).

### smb2.name-mangling.mangle / mangled-mask
**Error:** `Mangle names failed` and `NT_STATUS_NO_SUCH_FILE for mangled name`
**Cause:** 8.3 filename mangling is not implemented. When a long filename is created, ksmbd doesn't generate an 8.3 alias. Clients accessing the mangled name (e.g., `AEJEX~1.TXT`) get NOT_FOUND.
**Fix:** Implement 8.3 name mangling in the directory enumeration and name lookup path. Generate a deterministic 8.3 alias for filenames that exceed 8+3 characters and store/return it in `FindFileBothDirectoryInfo.ShortName`.

### smb2.mkdir.mkdir
**Error:** `NT_STATUS_ACCESS_DENIED should be NT_STATUS_OBJECT_PATH_SYNTAX_BAD`
**Cause:** Creating a directory with a malformed path (e.g., trailing backslash, double separators) is checked for access before the path is validated. The access check fires first and returns ACCESS_DENIED.
**Fix:** In `smb2_create.c`, validate path syntax (reject trailing separators, null components) before the access check. Return `STATUS_OBJECT_PATH_SYNTAX_BAD` for invalid path syntax.

## CATEGORY N: Server Bugs — Read / Write

### smb2.read.access
**Error:** `NT_STATUS_ACCESS_DENIED should be NT_STATUS_OK`
**Cause:** Reading metadata (stat open, DesiredAccess = FILE_READ_ATTRIBUTES) returns ACCESS_DENIED on a subsequent read. A stat-open handle should not be readable — this should return `STATUS_ACCESS_DENIED`. But the test expects OK — it opens with actual read access and the check is in the wrong place.
**Fix:** Review `smb2_read.c` access check — ensure that the daccess check correctly allows `FILE_READ_DATA` only when `FILE_READ_DATA` was in the granted access, without accidentally blocking valid reads.

### smb2.rw.append
**Error:** `NT_STATUS_ACCESS_DENIED should be NT_STATUS_OK`
**Cause:** Writing with `FILE_APPEND_DATA` (but not `FILE_WRITE_DATA`) at the sentinel offset `0xFFFFFFFFFFFFFFFF` (append-to-EOF) fails with ACCESS_DENIED. The access check in `smb2_write.c` is checking `FILE_WRITE_DATA_LE` exclusively for all writes, not accounting for `FILE_APPEND_DATA_LE` + sentinel combination.
**Fix:** In `smb2_write.c`, when offset == `0xFFFFFFFFFFFFFFFF` (the append sentinel), accept `FILE_APPEND_DATA_LE` as sufficient access in addition to `FILE_WRITE_DATA_LE`.

## CATEGORY O: Server Bugs — Replay / DHv2 Pending

### smb2.replay.dhv2-pending1n-vs-violation-lease-* (4 tests)
**Error:** `lb->new_lease_state got 0x3 (RH) should be 0x5 (RW)`
**Cause:** During DHv2 pending replay with a lease violation, the lease break notification sent to the client carries the wrong target state. The break should indicate `RW` (R+W, 0x5) but ksmbd sends `RH` (R+H, 0x3).
**Fix:** In the DHv2 pending replay path (`smb2_create.c` and `oplock.c`), when computing the break notification target state for the conflicting lease, compute it as `(current_state & ~SMB2_LEASE_HANDLE_CACHING_LE)` to remove H but keep W.

### smb2.replay.dhv2-pending1n-vs-oplock-sane / -windows
**Error:** `NT_STATUS_OK should be NT_STATUS_FILE_NOT_AVAILABLE`
**Cause:** A DHv2 replay in "pending" state (the original create is still in progress) against an oplock conflict should return `STATUS_FILE_NOT_AVAILABLE`. ksmbd returns OK.
**Fix:** In the DHv2 replay detection path, when the previous create is still pending (in-flight) and there's an oplock conflict, return `STATUS_FILE_NOT_AVAILABLE`.

### smb2.replay.dhv2-pending1n-vs-lease-sane / -windows
**Error:** `NT_STATUS_IO_TIMEOUT should be NT_STATUS_FILE_NOT_AVAILABLE`
**Cause:** Same as above but against a lease conflict. The request times out instead of returning `STATUS_FILE_NOT_AVAILABLE`.
**Fix:** In the DHv2 pending replay + lease conflict path, detect the pending state and return `STATUS_FILE_NOT_AVAILABLE` without waiting.
