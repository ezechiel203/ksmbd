# Delivery Sequence

## Purpose
Give a pragmatic order of execution so the project ships value before it takes on strategic parity scope.

## Sequence
### Step 1
Complete Program A planning and scope freeze.

### Step 2
Finish transport hardening and durable-state correctness.

### Step 3
Stand up the Program A interoperability lab early.

### Step 4
Qualify Hyper-V VM-disk scenarios on a bounded matrix.

### Step 5
Ship Program A with narrow, evidence-based claims.

### Step 6
Make a deliberate Program B go/no-go decision.

### Step 7
If Program B is approved, freeze NTFS, RSVD/VHDX, and failover architecture choices before implementation branches multiply.

### Step 8
Only then expand into NTFS-visible metadata, RSVD/shared-disk, or cluster-grade claims.

## Anti-Patterns
Do not:
1. start RSVD before Program A is stable
2. start NTFS parity work without architecture ownership
3. claim cluster support before a failover state model exists
4. broaden support claims based on one-off manual tests
