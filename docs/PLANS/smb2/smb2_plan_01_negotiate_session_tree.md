## Confirmed Bugs (P1)

### BUG-01: ServerGUID is always zeroed — client identity guarantee broken

**File:** `src/protocol/smb2/smb2_negotiate.c`
**Lines:** 888–891

```c
/* Not setting conn guid rsp->ServerGUID, as it
 * not used by client for identifying server
 */
memset(rsp->ServerGUID, 0, SMB2_CLIENT_GUID_SIZE);
```

**Symptom:** MS-SMB2 §2.2.4 requires `ServerGUID` to be "a globally unique identifier (GUID) that is generated by the server to uniquely identify this server". A client using multichannel (§3.2.5.2) or durable-handle reconnect may compare ServerGUIDs across connections to verify it is talking to the same server. Returning all-zeroes causes:
1. Any client that validates the GUID on reconnect will fail with `STATUS_CONNECTION_DISCONNECTED`.
2. Windows clients log event ID 30804 and may silently refuse to use multichannel.

**Spec ref:** MS-SMB2 §2.2.4 (ServerGUID field), §3.2.5.2 (multichannel server identity check).

**Fix:** Generate a random GUID once at module load (e.g. in `server_queue_ctrl_init_work`) and store it in `server_conf`. Emit it unchanged in every negotiate response.

```c
/* in server.h / server_conf */
u8 server_guid[SMB2_CLIENT_GUID_SIZE];

/* in server.c init path */
get_random_bytes(server_conf.server_guid, SMB2_CLIENT_GUID_SIZE);

/* in smb2_negotiate.c:891, replace memset with: */
memcpy(rsp->ServerGUID, server_conf.server_guid, SMB2_CLIENT_GUID_SIZE);
```

### BUG-02: SMB2.0 MaxTransactSize / MaxReadSize / MaxWriteSize advertised at 65536 bytes — violates SMB2.0 spec minimum for large MTU clients

**File:** `src/protocol/smb2/smb2ops.c`
**Lines:** 20–23

```c
static struct smb_version_values smb20_server_values = {
    .max_read_size  = CIFS_DEFAULT_IOSIZE,   /* 64 * 1024 = 65536 */
    .max_write_size = CIFS_DEFAULT_IOSIZE,
    .max_trans_size = CIFS_DEFAULT_IOSIZE,
```

**Symptom:** MS-SMB2 §2.2.4 states that for dialect 0x0202 the server SHOULD advertise `MaxTransactSize`, `MaxReadSize`, `MaxWriteSize` of at least 65536. The value 65536 (`0x10000`) is used here; this is exactly the minimum. However, the real Windows Server 2008 R2 (which introduced 0x0202) advertises 65536 for these fields and only SMB2.1+ advertises 1 MiB or 4 MiB. **This specific value is actually correct per the spec** — this is not a defect but is documented here for completeness because it does deviate from the higher values set for 2.1+.

**Status:** Acceptable — no change needed for compliance. Upgrading to 1 MiB would be a P4 performance optimisation.

### BUG-03: `SMB2_SESSION_REQ_FLAG_ENCRYPT_DATA` (0x04) in SESSION_SETUP request is never checked

**File:** `src/protocol/smb2/smb2_session.c`
**Lines:** 535–840 (entire `smb2_sess_setup()`)

**Symptom:** MS-SMB2 §2.2.5 defines flag `SMB2_SESSION_REQ_FLAG_ENCRYPT_DATA = 0x04`. When set by the client, §3.3.5.5.2 requires the server to set `SessionFlags.SMB2_SESSION_FLAG_ENCRYPT_DATA` in the response and encrypt all subsequent traffic on this session. The flag is defined at `smb2pdu.h:429` but is **never read** in `smb2_sess_setup()`. The server only sets `sess->enc = true` based on `smb3_encryption_negotiated(conn)` and the global `KSMBD_GLOBAL_FLAG_SMB2_ENCRYPTION` flag.

**Impact:** A client that sets `SMB2_SESSION_REQ_FLAG_ENCRYPT_DATA` expecting per-session encryption will receive unencrypted traffic, which is a correctness and security defect. Windows 11 with SMB encryption required will refuse to proceed.

**Spec ref:** MS-SMB2 §2.2.5 (Flags field), §3.3.5.5.2 ("If Connection.ClientCapabilities includes SMB2_GLOBAL_CAP_ENCRYPTION and the request Flags field has the SMB2_SESSION_REQ_FLAG_ENCRYPT_DATA bit set...").

**Fix:** In `ntlm_authenticate()` and `krb5_authenticate()`, after the encryption key has been generated, check `req->Flags & SMB2_SESSION_REQ_FLAG_ENCRYPT_DATA`. If set: force `sess->enc = true` and set `rsp->SessionFlags = SMB2_SESSION_FLAG_ENCRYPT_DATA_LE` regardless of the global flag.

### BUG-04: ServerStartTime is always set to 0 — per-spec it must reflect boot/start time

**File:** `src/protocol/smb2/smb2_negotiate.c`
**Line:** 894

```c
rsp->ServerStartTime = 0;
```

Also at `src/protocol/smb2/smb2_pdu_common.c:345` (SMB1 negotiate path).

**Symptom:** MS-SMB2 §2.2.4 defines `ServerStartTime` as "The time when the server was started, in FILETIME format". Setting it to 0 violates the spec. While most clients do not use this field for critical decisions, the Windows SMB client logs a warning when this is zero and some diagnostic tools flag it as an anomaly.

**Spec ref:** MS-SMB2 §2.2.4 (ServerStartTime).

**Fix:** Record the server start time (in 100-ns Windows FILETIME units) when the server initialises — e.g. in `server_queue_ctrl_init_work()` — and emit it in the negotiate response.

```c
/* server_conf addition */
u64 server_start_time_filetime;   /* set once at startup */

/* smb2_negotiate.c:894 */
rsp->ServerStartTime = cpu_to_le64(server_conf.server_start_time_filetime);
```

### BUG-05: LOGOFF does not return `STATUS_NETWORK_NAME_DELETED` on subsequent requests using the dead SessionId

**File:** `src/protocol/smb2/smb2_tree.c`
**Lines:** 411–416

```c
down_write(&conn->session_lock);
down_write(&sess->state_lock);
sess->state = SMB2_SESSION_EXPIRED;
up_write(&sess->state_lock);
up_write(&conn->session_lock);

ksmbd_all_conn_set_status(sess_id, KSMBD_SESS_NEED_SETUP);
```

**Symptom:** After LOGOFF the session object is marked `SMB2_SESSION_EXPIRED` and `KSMBD_SESS_NEED_SETUP`. MS-SMB2 §3.3.5.6 states: "The server MUST cancel all outstanding requests associated with this session." and "...any subsequent requests on this session MUST be failed with STATUS_USER_SESSION_DELETED." The session is put on the expired list but is not immediately removed from the session table. Depending on timing, a subsequent request that arrives in-flight before `ksmbd_user_session_put` frees the object can still be processed.

More critically, the spec requires that `STATUS_USER_SESSION_DELETED` (not `STATUS_NETWORK_NAME_DELETED`) be returned for post-LOGOFF requests. The error at `smb2_tree.c:408` says `STATUS_NETWORK_NAME_DELETED` which is the wrong status for this situation.

**Spec ref:** MS-SMB2 §3.3.5.6 step 6.

**Fix:** Change line 408 from `STATUS_NETWORK_NAME_DELETED` to `STATUS_USER_SESSION_DELETED`. The `ksmbd_check_user_session()` path already returns `STATUS_USER_SESSION_DELETED` for unknown session IDs, so the primary fix is ensuring the error code at the LOGOFF error path itself is correct.

## Missing Features (P2 — high interop impact)

### P2-01: SMB2_NETNAME_NEGOTIATE_CONTEXT_ID (type 5) is decoded but never validated

**File:** `src/protocol/smb2/smb2_negotiate.c`
**Lines:** 627–629

```c
} else if (pctx->ContextType == SMB2_NETNAME_NEGOTIATE_CONTEXT_ID) {
    ksmbd_debug(SMB,
                "deassemble SMB2_NETNAME_NEGOTIATE_CONTEXT_ID context\n");
```

**Description:** MS-SMB2 §2.2.3.1.4 specifies `SMB2_NETNAME_NEGOTIATE_CONTEXT_ID` carries the server name the client used to connect (in Unicode). §3.3.5.4 step 4 requires the server to: "Validate that the server name field in the `SMB2_NETNAME_NEGOTIATE_CONTEXT_ID` context matches one of the server's valid network names." If validation fails, the server SHOULD respond with `STATUS_OBJECT_NAME_NOT_FOUND`.

The current code silently ignores the contents. This breaks scenarios where a client connects to `SERVER-A` but the DNS resolves to a machine that was previously `SERVER-B` — it would succeed when it should fail.

**Spec ref:** MS-SMB2 §2.2.3.1.4, §3.3.5.4.

**Implementation approach:**
1. Extract the `NetName` Unicode field from the context.
2. Convert it to UTF-8 using `smb_strndup_from_utf16()`.
3. Compare it (case-insensitively) against `ksmbd_netbios_name()` and all interface addresses.
4. If no match, set `rsp->hdr.Status = STATUS_OBJECT_NAME_NOT_FOUND` and return `-EINVAL`.

### P2-02: Session binding signature verification uses `check_sign_req` which may accept unsigned compound requests

**File:** `src/protocol/smb2/smb2_session.c`
**Lines:** 678–685

```c
if (conn->binding) {
    if (!conn->ops->check_sign_req ||
        !conn->ops->check_sign_req(work)) {
        pr_err_ratelimited("Session binding signature verification failed\n");
        rc = -EACCES;
        goto out_err;
    }
}
```

**Description:** MS-SMB2 §3.3.5.5.2 says "The server MUST verify the signature of the request using Session.SigningKey." The `check_sign_req()` function (mapped to `smb3_check_sign_req`) verifies the SMB2 signature if the `SMB2_FLAGS_SIGNED` bit is present. However, if the binding SESSION_SETUP arrives as part of a compound request where only the first PDU is signed (related operations), the signature check on the binding PDU may pass vacuously (the first message signature covers the whole compound). The spec requires each SESSION_SETUP binding to be individually signed.

**Spec ref:** MS-SMB2 §3.3.5.2.7.

**Implementation approach:** When `conn->binding == true`, after `work->sess` is set, verify that `req->hdr.Flags & SMB2_FLAGS_SIGNED` is set and that `smb3_check_sign_req()` succeeds against the session's `smb3signingkey` directly, not a channel key. Currently `smb3_check_sign_req` dispatches correctly for non-binding cases; add an explicit check for binding that uses `sess->smb3signingkey`.

### P2-03: SMB2 TREE_CONNECT — MaximalAccess is a static bitmask, not computed from actual filesystem permissions

**File:** `src/protocol/smb2/smb2_tree.c`
**Lines:** 223–234

```c
rsp->ShareType = SMB2_SHARE_TYPE_DISK;
rsp->MaximalAccess = FILE_READ_DATA_LE | FILE_READ_EA_LE |
    FILE_EXECUTE_LE | FILE_READ_ATTRIBUTES_LE;
if (test_tree_conn_flag(status.tree_conn,
                        KSMBD_TREE_CONN_FLAG_WRITABLE)) {
    rsp->MaximalAccess |= FILE_WRITE_DATA_LE |
        FILE_APPEND_DATA_LE | FILE_WRITE_EA_LE |
        FILE_DELETE_LE | FILE_WRITE_ATTRIBUTES_LE |
        ...
```

**Description:** MS-SMB2 §2.2.10 states `MaximalAccess` is "the maximal access for the user on this share." The current implementation returns a hard-coded read-only or read-write mask based purely on the share's writable flag. It does not check the Linux filesystem permissions of the share root directory for the authenticated user. A guest user or a user with `read list` configured will receive the same `MaximalAccess` as an admin if `KSMBD_TREE_CONN_FLAG_WRITABLE` is set for the share.

Windows clients use `MaximalAccess` to pre-compute whether operations will succeed (e.g., `CreateFile` with `GENERIC_WRITE`). Returning an inflated `MaximalAccess` causes the client to attempt writes that will fail at the VFS layer, resulting in confusing `STATUS_ACCESS_DENIED` errors after the TREE_CONNECT succeeds.

**Spec ref:** MS-SMB2 §2.2.10, §3.3.5.7 step 6.

**Implementation approach:**
1. After share lookup succeeds, call `vfs_path_lookup()` on the share root.
2. Use `inode_permission()` (or `security_inode_permission()`) with `MAY_READ`, `MAY_WRITE`, `MAY_EXEC` against the authenticated user credentials.
3. Map Linux permissions to the SMB2 access mask.

### P2-04: TREE_CONNECT — PRINT share type (0x03) is never returned

**File:** `src/protocol/smb2/smb2_tree.c`
**Lines:** 213–235

The share type logic only produces `SMB2_SHARE_TYPE_PIPE` (0x02) or `SMB2_SHARE_TYPE_DISK` (0x01). `SMB2_SHARE_TYPE_PRINT` (0x03) is defined in `smb2pdu.h:505` but is never set. If ksmbd were to support print shares, the share type would need to be set to 0x03 and `ShareFlags` would need `SMB2_SHAREFLAG_PRINT_CACHING` (not currently defined).

**Spec ref:** MS-SMB2 §2.2.10 (ShareType).

**Implementation approach:** Add a `KSMBD_SHARE_FLAG_PRINT` bit to `ksmbd_netlink.h` alongside `KSMBD_SHARE_FLAG_PIPE`. Check it in `smb2_tree_connect()` and set `rsp->ShareType = SMB2_SHARE_TYPE_PRINT`. This is low priority because print support is otherwise absent.

### P2-05: LOGOFF does not explicitly cancel outstanding async requests for the session

**File:** `src/protocol/smb2/smb2_tree.c`
**Lines:** 393–403

```c
ksmbd_conn_lock(conn);
...
ksmbd_all_conn_set_status(sess_id, KSMBD_SESS_NEED_RECONNECT);
ksmbd_conn_unlock(conn);

smb2_send_session_closed_notification(work);

ksmbd_close_session_fds(work);
ksmbd_conn_wait_idle(conn);
```

**Description:** MS-SMB2 §3.3.5.6 step 3: "The server MUST cancel all outstanding requests." `ksmbd_conn_wait_idle()` waits for the reference count on `conn` to reach 1, which means all in-flight requests on this connection have completed. However, in an SMB3 multichannel scenario, the session spans multiple connections. Outstanding async requests on other channels (connections) for this session are not explicitly cancelled — they drain naturally only when those connections idle. The spec requires explicit cancellation.

**Spec ref:** MS-SMB2 §3.3.5.6 step 3.

**Implementation approach:** After `ksmbd_all_conn_set_status(sess_id, KSMBD_SESS_NEED_RECONNECT)`, iterate `sess->ksmbd_chann_list` and call `ksmbd_conn_wait_idle()` on each channel connection, or send a session expiry wakeup to each channel's worker thread.

## Partial Implementations (P3 — medium)

### P3-01: Pre-authentication integrity hash — NEGOTIATE hash updated, SESSION_SETUP hash updated, but final hash is not explicitly verified against session key derivation input

**File:** `src/protocol/smb2/smb2_session.c:78–113` (`generate_preauth_hash`)
**File:** `src/core/auth.c:1350–1378` (`ksmbd_gen_smb311_signingkey`)

**Description:** The pre-auth integrity (PAI) hash is correctly initialised from the negotiate hash at session allocation (`alloc_preauth_hash()`, line 70), and updated with each SESSION_SETUP request (`generate_preauth_hash()`, line 112). The final hash is correctly fed into signing key derivation (auth.c:1372: `d.context.iov_base = sess->Preauth_HashValue`) and encryption key derivation (auth.c:1438–1444).

However, MS-SMB2 §3.3.5.5.3 step 4 states that the *response* SESSION_SETUP PDU (with `STATUS_SUCCESS`) should also be hashed before being used as the signing key input. Currently, the response is not hashed — the signing key is derived using the hash state as it stands after the final *request* MESSAGE was hashed. The result is that the signing/encryption keys differ from what a strict spec-compliant client would derive.

**Evidence:** In `smb2_sess_setup()` (smb2_session.c:720), `generate_preauth_hash(work)` is called before the authentication handlers, which hash the request. After `ntlm_authenticate()` returns success (line 755), `sess->state = SMB2_SESSION_VALID` is set. But `ksmbd_gen_smb311_signingkey()` is called inside `ntlm_authenticate()` at line 411–416, before `ksmbd_iov_pin_rsp()` has written the response. The response PDU is therefore never hashed into the final PAI value.

**Spec ref:** MS-SMB2 §3.3.5.5.3 step 4; §3.1.4.1 ("The hash is also updated using the response message in the session setup exchange").

**Impact:** Interoperability issue with strict SMB 3.1.1 clients (rare in practice because Windows clients derive the key the same way — the spec wording is ambiguous about whether the server needs to hash the response before deriving keys vs. after sending it). Medium priority.

**Fix approach:** After `ksmbd_iov_pin_rsp()` creates the final SESSION_SETUP response buffer, call `ksmbd_gen_preauth_integrity_hash()` one more time on the response buffer, then call `generate_signingkey` and `generate_encryptionkey` using that final hash value.

### P3-02: Compression context — LZNT1, LZ77, LZ77+Huffman listed as unsupported but gracefully ignored

**File:** `src/protocol/smb2/smb2_negotiate.c:395–415` (`decode_compress_ctxt`)

**Description:** The server negotiates LZ4 or Pattern_V1 if offered by the client. If neither is offered, `conn->compress_algorithm` remains `SMB3_COMPRESS_NONE` and compression is silently disabled (status `STATUS_SUCCESS` returned). LZNT1, LZ77, and LZ77+Huffman are defined in `smb2pdu.h:312–315` but not matched in the algorithm selection loop. This is correct behavior per MS-SMB2 §3.3.5.4 (the server picks the best algorithm it supports), but the codebase lacks comments indicating *why* these are not supported, and the actual compression/decompression code for LZ77 variants is absent from `smb2_compress.c`.

The partial state: negotiation gracefully declines; actual compression is unimplemented for LZNT1/LZ77 variants. The `Flags` field in the `smb2_compression_ctx` (chained vs. unchained) is set to 0 unconditionally in `build_compress_ctxt()` (smb2_negotiate.c:92) — chained compression (flag = 1) is never offered even if client requests it.

**Spec ref:** MS-SMB2 §2.2.3.1.3, §3.3.5.4.

### P3-03: TREE_CONNECT extension (SMB 3.1.1 flag `EXTENSION_PRESENT`) — path offset adjusted but extension payload ignored

**File:** `src/protocol/smb2/smb2_tree.c:138–155`

**Description:** The `SMB2_TREE_CONNECT_FLAG_EXTENSION_PRESENT` (0x0004) bit in `req->Reserved` is checked. When present, the path offset is interpreted as relative to `Buffer[0]`, which is correct per MS-SMB2 §2.2.9. However, the `TREE_CONNECT_Request_Extension` structure itself (defined at §2.2.9.1) is never parsed. It contains `PathOffset`, `PathLength`, `TreeConnectContextOffset`, `TreeConnectContextCount`, and an array of `TREE_CONNECT_CONTEXT` elements. The important context types include:

- `SMB2_REMOTED_IDENTITY_TREE_CONNECT_CONTEXT_TYPE` (0x0001): remoted identity for impersonation.

The code extracts `path_ptr` but never reads `TreeConnectContextOffset` or `TreeConnectContextCount`, so tree connect contexts are silently dropped.

**Spec ref:** MS-SMB2 §2.2.9.1, §3.3.5.7 step 5.

### P3-04: TREE_CONNECT — `SMB2_TREE_CONNECT_FLAG_CLUSTER_RECONNECT` and `SMB2_TREE_CONNECT_FLAG_REDIRECT_TO_OWNER` are defined but never acted upon

**File:** `src/include/protocol/smb2pdu.h:498–499`
**File:** `src/protocol/smb2/smb2_tree.c:244–252` (Capabilities/ShareFlags building)

**Description:** The constants are defined:
```c
#define SMB2_TREE_CONNECT_FLAG_CLUSTER_RECONNECT  cpu_to_le16(0x0001)
#define SMB2_TREE_CONNECT_FLAG_REDIRECT_TO_OWNER  cpu_to_le16(0x0002)
```

**Spec ref:** MS-SMB2 §2.2.9, §2.2.10, §3.3.5.7.

### P3-05: TREE_DISCONNECT does not confirm outstanding async operations on the tree are cancelled before response

**File:** `src/protocol/smb2/smb2_tree.c:311–363`

**Description:** `ksmbd_close_tree_conn_fds(work)` closes all non-durable file handles. However, there may be outstanding async IOCTL, READ, WRITE, or CHANGE_NOTIFY operations still running in the work queue for this `tcon`. The TREE_DISCONNECT response is sent immediately after `ksmbd_tree_conn_disconnect()` without waiting for those operations to drain. MS-SMB2 §3.3.5.8 step 4 says "Cancel all outstanding requests that have `Request.TreeConnect` equal to `TreeConnect`". The current code marks `tcon->t_state = TREE_DISCONNECTED` (line 341) which causes subsequent requests using this TID to fail in `smb2_get_ksmbd_tcon()`, but in-flight requests that have already acquired a tcon reference continue to execute.

**Spec ref:** MS-SMB2 §3.3.5.8 step 4.

### P3-07: SESSION_SETUP — re-authentication (`SMB2_SESSION_VALID` re-entry) does not update signing/encryption keys

**File:** `src/protocol/smb2/smb2_session.c:361–367`

```c
down_read(&sess->state_lock);
if (sess->state == SMB2_SESSION_VALID) {
    up_read(&sess->state_lock);
    if (conn->binding)
        goto binding_session;
    return 0;           /* <-- returns without updating keys */
}
```

**Description:** MS-SMB2 §3.3.5.5.2 describes re-authentication: when a SESSION_SETUP completes on a `SMB2_SESSION_VALID` session (non-binding), the server should update `Session.SessionKey` and regenerate the signing and encryption keys. The current code returns 0 without calling `generate_signingkey` or `generate_encryptionkey`. This means a re-authenticated session retains its old keys, which is a security concern if the user changed their password.

**Spec ref:** MS-SMB2 §3.3.5.5.2, §3.3.5.5.3.

### P4-02: SMB2.0 capabilities advertised as 0 — correct but DFS capability not conditionally added

**File:** `src/protocol/smb2/smb2ops.c:20` — `smb20_server_values.capabilities = 0`
**File:** `src/protocol/smb2/smb2ops.c:261` (`init_smb2_1_server` adds DFS for SMB2.1)

SMB2.0 does not add `SMB2_GLOBAL_CAP_DFS` even when DFS is enabled (`ksmbd_dfs_enabled()`), unlike SMB2.1+ (line 262). MS-SMB2 §2.2.4 says SMB2.0 servers should still set `SMB2_GLOBAL_CAP_DFS` if they support DFS. Low impact as SMB2.0 is rarely used in DFS scenarios.

### P4-03: Compression `Flags` field (chained vs. unchained) is hardcoded to 0

**File:** `src/protocol/smb2/smb2_negotiate.c:92`

```c
pneg_ctxt->Flags = cpu_to_le32(0);   /* always "unchained" */
```

MS-SMB2 §2.2.3.1.3 defines bit 0 of `Flags` as `SMB2_COMPRESSION_CAPABILITIES_FLAG_CHAINED` (0x00000001). The server never advertises chained compression support. This is acceptable given that chained compression decompression is not implemented in `smb2_compress.c`.

### P4-04: `SIGNING_ALG_HMAC_SHA256` (0x0000) priority in signing capability selection

**File:** `src/protocol/smb2/smb2_negotiate.c:460–470`

When a client offers only `SIGNING_ALG_HMAC_SHA256` and not AES algorithms, the server accepts it (priority 0 > -1). This is correct. When a SMB3 client uses HMAC-SHA256 signing, the `smb3_check_sign_req` path handles it correctly because `check_sign_req` dispatches through the ops table. No defect — documented for completeness.

### P4-05: The `NUMBER_OF_SMB2_COMMANDS` constant (0x0013 = 19) does not include `SMB2_SERVER_TO_CLIENT_NOTIFICATION` (0x0013)

**File:** `src/include/protocol/smb2pdu.h:110`

```c
#define NUMBER_OF_SMB2_COMMANDS  0x0013
```

`SMB2_SERVER_TO_CLIENT_NOTIFICATION_HE = 0x0013` is defined at line 47 but the command table `smb2_0_server_cmds` is sized `NUMBER_OF_SMB2_COMMANDS = 19 = 0x0013`, meaning the notification slot is actually index 19 (0x0013) and the array is 0-indexed 0–18. The `smb2_notify` dispatch is registered at `SMB2_CHANGE_NOTIFY_HE = 0x000F` (slot 15), not at 0x0013. The notification is server-initiated, so it is never dispatched through the incoming command table. No runtime bug, but the constant is confusingly named.
