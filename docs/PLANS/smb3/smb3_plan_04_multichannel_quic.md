## Multichannel (Session Binding) Audit

### Feature Gate

`KSMBD_GLOBAL_FLAG_SMB3_MULTICHANNEL` controls the feature at both
advertising and binding time.

**Capability advertisement** (`src/protocol/smb2/smb2ops.c`, lines 293-294,
328-329, 361-362):
```c
if (server_conf.flags & KSMBD_GLOBAL_FLAG_SMB3_MULTICHANNEL)
    conn->vals->capabilities |= SMB2_GLOBAL_CAP_MULTI_CHANNEL;
```
This is set for SMB 3.0, 3.0.2, and 3.1.1 server capability structures.
Correct: the flag is only set when the feature is enabled in config, and
only for dialect >= SMB 3.0 as required by MS-SMB2 §2.2.3.

### SESSION_SETUP Binding Path

`src/protocol/smb2/smb2_session.c`, lines 572–636.

The binding decision tree:

```
if !req->hdr.SessionId:          → new session, conn->binding = false
elif dialect >= SMB30 AND multichannel AND BINDING flag set:
    → session binding path
elif dialect < SMB30 AND BINDING flag:
    → STATUS_REQUEST_NOT_ACCEPTED (correct per spec)
else:
    → re-auth / new session on existing conn
```

**Binding path checks (lines 575–636):**

1. **Session lookup across all connections** — `ksmbd_session_lookup_slowpath(sess_id)`:
   correct; looks up by session ID globally, not just on this connection.

2. **Dialect match** — `conn->dialect != sess->dialect` → `-EINVAL`: correct.

3. **Signature required** — Lines 594–597 check `SMB2_FLAGS_SIGNED` is set
   before proceeding. Correct per MS-SMB2 §3.3.5.2.7.

4. **ClientGUID match** — `memcmp(conn->ClientGUID, sess->ClientGUID, ...)`:
   correct per MS-SMB2 §3.3.5.5.3.

5. **Session-in-progress guard** — `SMB2_SESSION_IN_PROGRESS` returns
   `STATUS_REQUEST_NOT_ACCEPTED` (via `-EACCES`). Correct.

6. **Session-expired guard** — `SMB2_SESSION_EXPIRED` returns
   `STATUS_NETWORK_SESSION_EXPIRED` (via `-EFAULT`). Correct.

7. **Guest sessions rejected** — `user_guest(sess->user)` → `-EOPNOTSUPP`:
   correct, guest sessions cannot be extended via multichannel.

8. **Duplicate channel guard** — `is_ksmbd_session_in_connection(conn, sess_id)`:
   prevents binding the same connection twice to the same session. Correct.

9. **Binding signature verification** — Lines 678–685: after `work->sess`
   is set, the binding SESSION_SETUP is signature-verified with the existing
   session's signing key via `conn->ops->check_sign_req(work)`. Correct
   deferral of verification so `work->sess` is available.

10. **SessionId in response** — The response reuses `req->hdr.SessionId`
    because it is not reset in the binding branch; `rsp->hdr.SessionId` is
    inherited from the SMB2 header copy. Correct: the existing SessionId
    must be returned unchanged.

### Channel Table Management

`struct ksmbd_session` (`src/mgmt/user_session.h`, line 59):
```c
struct xarray ksmbd_chann_list;
```
The channel list is an XArray keyed by `(long)conn`. Each `struct channel`
stores the connection pointer and the per-channel signing key:
```c
struct channel {
    __u8            smb3signingkey[SMB3_SIGN_KEY_SIZE];
    struct ksmbd_conn *conn;
};
```

Channel insertion (`src/protocol/smb2/smb2_session.c`, lines 394–408 and
500–515):
```c
chann = kmalloc(sizeof(struct channel), ...);
chann->conn = conn;
xa_store(&sess->ksmbd_chann_list, (long)conn, chann, ...);
```

`lookup_chann_list()` (`src/protocol/smb2/smb2_pdu_common.c`, line 118):
```c
return xa_load(&sess->ksmbd_chann_list, (long)conn);
```

### Per-Channel Signing

Per-channel signing keys are generated by `generate_signingkey()` called
at the end of both `ntlm_authenticate()` and `krb5_authenticate()` after
the channel is inserted. `src/core/auth.c` line 1315 looks up the channel
and stores the derived key in `chann->smb3signingkey`. Correct per
MS-SMB2 §3.3.5.5.3: each channel has its own signing key derived from
the session master key and the channel's pre-authentication integrity hash.

For sign-verification (`src/protocol/smb2/smb2_pdu_common.c`, line 1091):
```c
chann = lookup_chann_list(work->sess, conn);
signing_key = chann->smb3signingkey;
```
For SESSION_SETUP responses the session-level `smb3signingkey` is used;
for all other commands the per-channel key is used. Correct.

### Per-Channel Encryption State

There is **no per-channel encryption key**. Encryption uses the session-level
keys (`sess->smb3encryptionkey` / `sess->smb3decryptionkey`). This is correct
per the spec: encryption keys are session-scoped, not channel-scoped. ✓

### Per-Channel Credit Tracking

Credit tracking is **per-connection**, not per-channel
(`struct ksmbd_conn.total_credits`, `.outstanding_credits`). This is
architecturally correct: each underlying TCP (or QUIC) connection is an
independent transport with its own credit window. The spec's credit model
in MS-SMB2 §3.2.4.1.5 operates per-connection, not per-session-channel.
The current implementation correctly reflects this. ✓

### ChannelSequence Validation

`smb2_check_channel_sequence()` (`src/protocol/smb2/smb2_pdu_common.c`,
line 86) implements MS-SMB2 §3.3.5.2.10 ChannelSequence validation for
WRITE, LOCK, IOCTL, SET_INFO operations. The signed-difference comparison
correctly handles wrap-around:
```c
diff = (s16)(req_seq - fp->channel_sequence);
if (diff < 0) return -EAGAIN;  // stale, return STATUS_FILE_NOT_AVAILABLE
if (diff > 0) fp->channel_sequence = req_seq;  // advance
```
Correct. Note: `FLUSH` is not explicitly listed in the call sites but FLUSH
does not carry a ChannelSequence in the header — it uses the Status field
differently. Confirmed no gap here.

### Session Binding Failure Cleanup

If binding fails (`rc < 0` at `out_err`), the session state is set to
`SMB2_SESSION_EXPIRED` and `ksmbd_user_session_put()` is called
(`src/protocol/smb2/smb2_session.c`, lines 813–822). The channel is **not**
inserted at this point because `binding_session:` is only reached after
successful authentication. If authentication fails before the channel is
inserted, there is nothing to remove. If authentication passes but then
signing verification fails, the channel has already been inserted. In this
case `out_err` sets the session expired but does not explicitly remove the
newly-inserted channel. This is a latent issue; however the session is
immediately marked expired and the next `ksmbd_sessions_deregister()` call
on connection teardown will clean up. Low-severity.

### Connection Teardown Under Multichannel

`ksmbd_sessions_deregister()` (`src/mgmt/user_session.c`, lines 271–338):

When a non-binding connection exits:
- The channel for that connection is erased from `ksmbd_chann_list`.
- If the channel list becomes empty, the session is removed from the global
  table and the session is destroyed.
- If other channels remain, the session survives. Other channel connections
  are signalled via `ksmbd_conn_set_exiting()` at line 323.

When a binding connection exits (line 293):
- All sessions that had a channel on this connection are iterated.
- Each channel is removed; if the session becomes channel-less it is destroyed.

This correctly implements MS-SMB2 §3.3.4.4 channel teardown semantics:
a session survives as long as any channel remains. ✓

**Session-closed notification to other channels:**
`smb2_send_session_closed_notification()` (`src/protocol/smb2/smb2_misc_cmds.c`,
lines 600–668) iterates `ksmbd_chann_list` and sends `SMB2_NOTIFY_SESSION_CLOSED`
to each peer channel connection when a session logoff is processed. Correct
per MS-SMB2 §3.3.4.1.8.
Minor: notification is only sent for `conn->dialect >= SMB311_PROT_ID`
(line 607). MS-SMB2 §3.3.4.1.8 requires notification for all multichannel
sessions, not just 3.1.1. Should be `>= SMB30_PROT_ID`.

### Durable Handles Across Channels

Durable handles (`fp->is_durable`, `fp->is_persistent`) are session-scoped
via the file table, not channel-scoped. When a channel fails the session
remains alive on other channels and the durable handles remain open. When
the *last* channel closes, the session is torn down and
`ksmbd_close_session_fds()` is called, which closes all file handles
including durable ones. This is consistent with MS-SMB2 §3.3.5.1 where
durable handles survive network disconnection only at the session level,
not across full session teardown. ✓

### Compound Requests Across Channels

Compound requests are assembled in the connection handler loop of a single
connection (`src/core/connection.c`, `ksmbd_conn_handler_loop()`). The
RFC1002 4-byte length framing ensures each PDU is read atomically by one
thread. There is no cross-channel compound assembly. Per MS-SMB2 §3.2.4.1.4
compounded requests MUST be sent on a single connection; the server never
needs to reassemble across channels. This is correct by design. ✓

## FSCTL_QUERY_NETWORK_INTERFACE_INFO Audit

Handler: `fsctl_query_iface_info_ioctl()`,
`src/protocol/smb2/smb2_ioctl.c`, lines 226–341.
(A duplicate implementation also exists in `src/fs/ksmbd_fsctl.c` lines
~830–920, presumably the registered FSCTL handler; the one in smb2_ioctl.c
is the legacy fallback.)

### NIC Enumeration

Iterates `init_net` with `for_each_netdev()`. Filters:
- Skips loopback (`ARPHRD_LOOPBACK`). ✓
- Only includes NICs present in `iface_list` via `ksmbd_find_netdev_name_iface_list()`.
  **This is a significant limitation**: only NICs explicitly configured with
  `ksmbd_tcp_set_interfaces()` are reported. If the server listens on all
  interfaces (`bind_additional_ifaces = 1`), the dynamically-discovered
  interfaces are added to `iface_list` at NETDEV_UP time, but NICs that
  were up before ksmbd started may not appear. This can cause the client
  to miss channels.
- Skips NICs with `!(IFF_RUNNING)`. ✓

### Response Structure

```c
nii_rsp->Next = cpu_to_le32(152);  // fixed stride of 152 bytes
nii_rsp->IfIndex = cpu_to_le32(netdev->ifindex);  ✓
nii_rsp->Capability = 0;           // RSS_CAPABLE | RDMA_CAPABLE conditionally set ✓
nii_rsp->LinkSpeed = cpu_to_le64(speed);  // 64-bit bytes/sec ✓
```

The last entry correctly sets `nii_rsp->Next = 0` (line 336). ✓

### LinkSpeed

LinkSpeed is reported in **bits per second × 1,000,000**
(line 285: `speed *= 1000000`).
`cmd.base.speed` from `get_link_ksettings()` is in Mbps.
So 1000 Mbps * 1,000,000 = 1,000,000,000,000 bps = 1 Tbps? That is wrong.

**BUG**: MS-SMB2 §2.2.32.4 specifies LinkSpeed in **bytes per second**.
`ethtool` speed is in Mbps (megabits per second).
Correct conversion: `speed_Mbps * 1000 * 1000 / 8` (bytes/sec).
Current code: `speed_Mbps * 1,000,000` = bits/sec (off by factor of 8 *and*
in the wrong unit).
Example: 1 GbE reports `1000 * 1,000,000 = 1,000,000,000` which happens to
be 10^9, but the unit is bits not bytes, making it 8x too high. The MS-SMB2
spec explicitly states this field is "LinkSpeed in bytes per second."

### SockAddr_Storage Port Field

Both IPv4 and IPv6 entries set `Port = 0`:
```c
sockaddr_storage->addr4.Port = 0;  // line 296
sockaddr_storage->addr6.Port = 0;  // line 312
```
MS-SMB2 §2.2.32.4 NETWORK_INTERFACE_INFO SockAddr_Storage:
"If the family is INTERNETWORK, the Port field MUST be set to 0x0000."
Port = 0 is actually correct per the spec for this response structure.
There is no requirement to report port 445 here. ✓

### RSS_CAPABLE Flag

```c
    nii_rsp->Capability |= cpu_to_le32(RSS_CAPABLE);
```
RSS detection via `real_num_tx_queues > 1` is a reasonable heuristic.

### RDMA_CAPABLE Flag

```c
    nii_rsp->Capability |= cpu_to_le32(RDMA_CAPABLE);
```

`ksmbd_rdma_capable_netdev()` is defined in `transport_rdma.c` line 2566

### IPv6 Address Enumeration

The `ipv6_retry:` label logic emits two entries per NIC: one IPv4, one IPv6.
The IPv6 entry skips tentative or deprecated addresses. ✓
However the `goto ipv6_retry` approach is somewhat fragile; if an interface

## QUIC Transport Audit

### Architecture

The QUIC transport uses a **userspace proxy** architecture:

```
SMB Client --[QUIC/TLS 1.3, port 443]--> [Userspace Proxy]
                                                 |
                                    [Abstract Unix Socket @ksmbd-quic]
                                                 |
                                     [Kernel ksmbd QUIC module]
```

The kernel module listens on abstract unix domain socket `@ksmbd-quic`,
accepts connections from the userspace proxy, reads a fixed
`ksmbd_quic_conn_info` header, then processes raw SMB2 PDU bytes.

### Listener

`create_unix_listener()` (`src/transport/transport_quic.c`, lines 625–675):
- Creates `AF_UNIX SOCK_STREAM` socket.
- Binds to abstract address `\0ksmbd-quic`.
- Listens with backlog 64.
- Spawns `ksmbd_quic_listener_fn` kthread.

The listener is always started unconditionally when `CONFIG_SMB_SERVER_QUIC`
is enabled (`src/core/connection.c`, line 776).

### TLS / QUIC Handshake

There is **no TLS 1.3 or QUIC implementation in the kernel module**.
TLS termination is entirely delegated to the userspace proxy. The kernel
sees only post-decryption SMB2 PDU bytes. The proxy communicates the result
of TLS verification via `conn_info.flags`:
```c
#define KSMBD_QUIC_F_TLS_VERIFIED  BIT(0)
#define KSMBD_QUIC_F_EARLY_DATA    BIT(1)
```
The kernel rejects connections that do not have `KSMBD_QUIC_F_TLS_VERIFIED`
set (`src/transport/transport_quic.c`, lines 393–397). ✓

### Port

The constant `KSMBD_QUIC_PORT = 443` is defined in
`src/include/transport/transport_quic.h` line 14, but it is not actually
used anywhere in the kernel module. The kernel only listens on a unix socket;
port 443 must be handled by the userspace proxy. The constant is
documentation/convention, not enforced by the kernel. This is architecturally
intentional but means port validation is entirely outside kernel control.

### Security: Proxy Authentication

Two layers of root-check are applied:
1. `ksmbd_quic_listener_fn()` lines 573–585: checks `sk_peer_cred->uid == 0`
   immediately after accept.
2. `verify_proxy_peer()` lines 78–110: redundant second check inside
   `ksmbd_quic_new_connection()`.

The double-check is redundant but not harmful. Using `uid == 0` as the sole
authentication criterion means any root process can inject SMB connections.
There is no certificate-based or capability-based access control. For a
production deployment this is a significant security concern.

### SMB2 Framing Over QUIC

The `ksmbd_conn_handler_loop()` in `connection.c` is shared across all
transport types. It reads the RFC1002 4-byte length prefix then the PDU body.
This is the NetBIOS Session Service framing, not the raw-SMB2 framing used
by some QUIC implementations. The userspace proxy must therefore also prepend
the 4-byte RFC1002 NetBIOS session header.

MS-SMB2-QUIC (RFC 9446 / MS-SMBD-QUIC draft) specifies that SMB over QUIC
uses SMB2 PDUs without the NetBIOS session wrapper. **The current
implementation retains the NetBIOS framing**, which is non-standard for
SMB over QUIC. This requires corresponding framing in the userspace proxy.
If the proxy connects directly to an SMB-over-QUIC client without re-framing,
the framing will mismatch.

### SMB2_TRANSPORT_CAPABILITIES Negotiation

`build_transport_cap_ctxt()` (`src/protocol/smb2/smb2_negotiate.c`, line 131):
```c
pneg_ctxt->Flags = SMB2_ACCEPT_TRANSPORT_LEVEL_SECURITY;
```
This context is sent only when `conn->transport_secured` is true (line 265),
which is set when the client sends `SMB2_ACCEPT_TRANSPORT_LEVEL_SECURITY` in
its own TRANSPORT_CAPABILITIES negotiate context
(`src/protocol/smb2/smb2_negotiate.c`, lines 491–497):
```c
if (pneg_ctxt->Flags & SMB2_ACCEPT_TRANSPORT_LEVEL_SECURITY)
    conn->transport_secured = true;
```
This is a mutual capability exchange: if client says it has transport-level
security, server echoes back. For QUIC connections this is appropriate since
TLS is provided at the transport layer. However the kernel does not
distinguish QUIC connections from TCP connections when setting
`transport_secured` — it relies solely on what the client advertises.
A QUIC proxy could set this correctly, but a malicious client on TCP could
also set it and bypass SMB-layer signing. This is a trust issue: the
transport-secured flag should be authoritative from the transport layer, not
from the client.

### 0-RTT (Early Data)

`KSMBD_QUIC_F_EARLY_DATA` flag is defined but never acted upon in the
kernel module. Early data replay protection is not implemented.

### IPv6 Client Address Handling (QUIC)

When the proxy reports `AF_INET6`, the kernel uses only the lower 32 bits
for hashing (`src/transport/transport_quic.c`, lines 452–454):
```c
memcpy(&conn->inet_addr, &t->conn_info.client_addr.v6[12], 4);
conn->inet_hash = ipv4_addr_hash(conn->inet_addr);
```
The full 128-bit IPv6 address is not stored in `conn->inet6_addr`, so
per-IP connection counting for IPv6 QUIC clients uses a truncated address.
This means IPv6 clients with the same lower 32 bits but different upper bits
share the connection count bucket, causing false-positive rate limiting.

### QUIC Transport Ops

```c
static const struct ksmbd_transport_ops ksmbd_quic_transport_ops = {
    .read           = ksmbd_quic_read,
    .writev         = ksmbd_quic_writev,
    .disconnect     = ksmbd_quic_disconnect,
    .free_transport = ksmbd_quic_free_transport,
};
```
Notable: no `.shutdown` op. When `stop_sessions()` iterates connections,
connections without a `.shutdown` op are not explicitly shut down (see
`src/core/connection.c` line 811). The QUIC transport relies solely on
`kernel_sock_shutdown()` in `free_transport()`. For graceful server teardown
this means QUIC connections may not be terminated promptly.

### Partial Write in `ksmbd_quic_writev`

`ksmbd_quic_writev()` (`src/transport/transport_quic.c`, lines 270–313)
handles partial sends by advancing the iov pointer in place. However it
mutates the caller's iov array directly (increments `iov` pointer and modifies
`iov->iov_base`/`iov->iov_len`). If the caller needs the iov again after
the call this would be a bug. For the current call path through
`ksmbd_conn_write()` → `writev()`, the iov is not reused, so this is not
currently a problem.

### Current State Assessment

The QUIC transport is **partial/stub**:
- Kernel-side listener: implemented and functional.
- SMB2 PDU handling: functional (via shared handler loop).
- TLS/QUIC: not in kernel, fully delegated to userspace proxy.
- Userspace proxy: **not provided in this repository**.
- Port 443 listener: not in kernel.
- RFC1002 framing: non-standard for SMB-over-QUIC spec.
- SMB_TRANSPORT_CAPABILITIES context: set but not authoritatively tied to
  transport type.

**Key missing pieces for end-to-end QUIC:**
1. Userspace proxy (msquic/quiche/ngtcp2 based) with TLS 1.3 certificate
   handling, port 443 listener, and unix socket forwarding.
2. Removal of RFC1002 NetBIOS framing (or proxy must add it).
3. Full IPv6 address storage in `ksmbd_conn` for QUIC connections.
4. `.shutdown` transport op to enable graceful teardown.
5. `0-RTT` replay protection if early data is to be supported.
6. Authoritative `transport_secured` flag set by transport layer, not client.

## RDMA Transport Summary

### Listener

`ksmbd_rdma_init()` → `smb_direct_listen()` (`src/transport/transport_rdma.c`,
lines 2370–2500). Listens on:
- Port 5445 (iWARP)
- Port 445 (InfiniBand, default `smb_direct_port = SMB_DIRECT_PORT_INFINIBAND`)

The port is selected at init time based on device type.

### Credit Management

Full SMBD credit protocol is implemented in `smb_direct_transport`:
```c
spinlock_t receive_credit_lock;
int recv_credits, recv_credit_max, recv_credit_target;
atomic_t send_credits;
int send_credit_target, new_recv_credits;
atomic_t rw_credits;
int max_rw_credits, pages_per_rw_credit;
```
Post-receive credit replenishment via `smb_direct_post_recv_credits` workqueue
item. Send-credit wait queue (`wait_send_credits`). This is a substantive
implementation of MS-SMBD §3.1.1.1 credit fields. ✓

### RDMA Capability Advertisement

`ksmbd_rdma_capable_netdev()` is called from `fsctl_query_iface_info_ioctl()`
and `src/fs/ksmbd_fsctl.c` to set `RDMA_CAPABLE` in the NIC enumeration
response. Correct. ✓

`SMB2_GLOBAL_CAP_RDMA` is not explicitly set in the negotiate capabilities
struct (`src/protocol/smb2/smb2ops.c` is not showing it). The RDMA capability
is not advertised in Capabilities flags; instead it is implied by SMBD
transport being active. Windows clients discover RDMA capability through
`FSCTL_QUERY_NETWORK_INTERFACE_INFO` response. This matches production
ksmbd behavior.

### RDMA Transform Capabilities

RDMA transform context negotiation is fully implemented: `decode_rdma_transform_ctxt()`
and `build_rdma_transform_ctxt()` are present in smb2_negotiate.c. ✓

### Missing for Full RDMA Compliance

1. `SMB_DIRECT_PORT_IWARP = 5445` vs `SMB_DIRECT_PORT_INFINIBAND = 445` —
   the default is 445 which conflicts with TCP. In practice kernel RDMA and
   TCP cannot both bind port 445, so RDMA is typically on port 5445 for
   iWARP. The current code switches ports at runtime based on detected
   device type (lines 2404–2407).
2. No explicit test for iWARP vs InfiniBand in `FSCTL_QUERY_NETWORK_INTERFACE_INFO`
   response — both are reported as `RDMA_CAPABLE` without distinguishing
   transport type.
3. RDMA-write path for large reads/writes (`rdma_read` / `rdma_write` ops)
   is implemented. Functional completeness cannot be determined without
   hardware testing.

## Confirmed Bugs (P1)

### BUG-MC-01: LinkSpeed unit error in FSCTL_QUERY_NETWORK_INTERFACE_INFO

**File:** `src/protocol/smb2/smb2_ioctl.c`, line 285
(also `src/fs/ksmbd_fsctl.c` ~line 868)

**Symptom:** `speed *= 1000000` converts `ethtool` Mbps to bits/sec, but
MS-SMB2 §2.2.32.4 requires **bytes/sec**. This makes LinkSpeed 8× too large.
A 1 GbE NIC reports `1,000,000,000` (= 10^9) which coincidentally looks
correct numerically but is in the wrong unit. A 10 GbE NIC reports
`10,000,000,000` (= 10^10) which is 8× too high (should be `1,250,000,000`
bytes/sec).

**Spec ref:** MS-SMB2 §2.2.32.4 NETWORK_INTERFACE_INFO, LinkSpeed field:
"A 64-bit unsigned integer indicating the speed of the network interface
in bytes per second."

**Fix:** `speed = (u64)cmd.base.speed * 1000 * 1000 / 8` (bytes/sec).

### BUG-MC-02: Session-closed notification limited to SMB 3.1.1+

**File:** `src/protocol/smb2/smb2_misc_cmds.c`, line 607 and line 621

**Symptom:**
```c
if (!sess || curr_conn->dialect < SMB311_PROT_ID)
    return;
...
if (ch_conn->dialect < SMB311_PROT_ID)
    continue;
```
SESSION_CLOSED notifications are only sent for 3.1.1 connections. A
multichannel session established over SMB 3.0 or 3.0.2 will not receive
session-closed notifications on peer channels.

**Spec ref:** MS-SMB2 §3.3.4.1.8: notification applies to all multichannel
sessions.

**Fix:** Change dialect guards from `SMB311_PROT_ID` to `SMB30_PROT_ID`.

### BUG-MC-03: IPv6 address truncation in QUIC per-IP limiting

**File:** `src/transport/transport_quic.c`, lines 452–458

**Symptom:** For QUIC connections from IPv6 clients, only the lower 32 bits
of the IPv6 address are stored in `conn->inet_addr`. The full 128-bit address
is not stored in `conn->inet6_addr`. The per-IP connection count check in
`ksmbd_quic_new_connection()` (lines 503–514) only compares `inet_addr`
(32-bit). Two IPv6 clients with the same lower 32 bits of address but
different upper 96 bits will share a connection count, causing false
rejection.

**Spec ref:** Not a spec violation per se, but a correctness/security bug.

**Fix:** Store full IPv6 address in `conn->inet6_addr` for QUIC IPv6
connections, and use `ipv6_addr_hash()` for `inet_hash`. Mirror the logic
used in the TCP transport (`src/transport/transport_tcp.c`, lines 129–135).

---

### BUG-MC-04: QUIC missing `.shutdown` transport op

**File:** `src/transport/transport_quic.c`, lines 730–735

**Symptom:** `ksmbd_quic_transport_ops` does not provide a `.shutdown` op.
In `stop_sessions()` (`src/core/connection.c`, lines 804–828), only
connections whose transport provides `.shutdown` are shut down gracefully:
```c
if (t->ops->shutdown) {
    t->ops->shutdown(t);
    ...
}
```
QUIC connections will never receive a graceful shutdown signal during server
stop, potentially hanging the `stop_sessions()` loop.

**Spec ref:** Operational correctness.

**Fix:** Implement a `ksmbd_quic_shutdown()` that calls
`kernel_sock_shutdown(t->sock, SHUT_RDWR)` and register it as `.shutdown`.

## Missing Features (P2)

### P2-MC-01: NIC enumeration misses pre-existing interfaces

**File:** `src/protocol/smb2/smb2_ioctl.c`, lines 238–247

When `bind_additional_ifaces = 1` (all interfaces mode), NICs that were up
before ksmbd loaded are not in `iface_list` because `ksmbd_netdev_event()`
only adds them dynamically on NETDEV_UP. An initial scan of current interfaces
at startup is missing. Clients may discover fewer channels than expected.

**Fix:** At `ksmbd_tcp_init()` time, enumerate all currently-up netdevs and
add them to `iface_list`.

### P2-MC-02: No QUIC userspace proxy provided

The kernel QUIC module is non-functional without a matching userspace proxy.
No proxy implementation is present in the repository. TLS certificate loading,
port 443 binding, ALPN negotiation (`smb`), and unix socket forwarding must
all be implemented in userspace.

### P2-MC-03: RFC1002 framing in QUIC violates MS-SMB2-QUIC spec

MS-SMB2-QUIC (RFC 9446 §3.1) specifies SMB2 PDUs are sent directly in QUIC
stream data without the 4-byte NetBIOS session wrapper. The current
implementation reads the RFC1002 header as the first 4 bytes
(`connection.c`, line 570). The userspace proxy must add synthetic RFC1002
framing, which is non-standard.

### P2-MC-04: Transport-secured flag not authoritatively set by kernel

`conn->transport_secured` is set based on what the client advertises in its
SMB2_TRANSPORT_CAPABILITIES negotiate context. For QUIC this is functionally
correct (client knows it has TLS), but the kernel cannot verify it. An SMB
client over plain TCP could falsely claim transport security, potentially
bypassing SMB signing enforcement. The transport layer should set this flag
authoritatively.

## Partial (P3)

### P3-MC-01: Channel insertion before signing-key derivation window

In both `ntlm_authenticate()` and `krb5_authenticate()`, the channel is
inserted into `ksmbd_chann_list` (lines 394–408 / 500–515) before the
signing key is derived (`generate_signingkey()` at lines 411–417 / 518–524).
There is a brief window where a channel exists in the list without a valid
signing key. Concurrent requests on other channels using `lookup_chann_list()`
could transiently find a channel with a zeroed signing key. Low risk due to
holding `conn->srv_mutex` for the duration of SESSION_SETUP, but worth noting.

### P3-MC-02: Session-level vs channel-level pre-auth hash for binding

The pre-authentication integrity hash for binding SESSION_SETUP is maintained
in a separate per-session `preauth_session` list (conn's `preauth_sess_table`),
not the session's main `Preauth_HashValue`. This is correct per MS-SMB2
§3.3.5.5.3, but the removal (`ksmbd_preauth_session_remove()`) only happens
for NTLMSSP authentication (line 757). For Kerberos binding the preauth
session entry is never explicitly removed, leaking a small amount of memory
per bound channel after session setup. `connection.c:ksmbd_conn_cleanup()`
does clean up the whole `preauth_sess_table`, so it is not a permanent leak.

### P3-MC-03: QUIC 0-RTT early data handling undefined

`KSMBD_QUIC_F_EARLY_DATA` flag is defined but the kernel takes no action
if it is set. If the proxy allows 0-RTT, replayed early data could create
duplicate SMB2 requests. The kernel has no defense against this.

## Low Priority (P4)

### P4-MC-01: Fixed `Next` stride of 152 bytes in NIC info response

`nii_rsp->Next = cpu_to_le32(152)` is hardcoded. `sizeof(struct network_interface_info_ioctl_rsp)` is 152. If the struct is ever extended this becomes wrong. Should use `cpu_to_le32(sizeof(*nii_rsp))`.

### P4-MC-02: Double proxy root-check

`verify_proxy_peer()` and the inline check in `ksmbd_quic_listener_fn()` both
verify `uid == 0`. One of them is redundant. Minor code duplication.

### P4-MC-03: `ksmbd_quic_conn_info` alignment

`struct ksmbd_quic_conn_info` is `__packed`. If future fields are added
without considering alignment, unaligned 32/64-bit reads may occur on
architectures that require alignment.

### P4-MC-04: ksmbd_quic_writev mutates caller's iov

`ksmbd_quic_writev()` advances the passed-in `iov` pointer and modifies
`iov_base`/`iov_len` in place. Caller must not reuse the iov after the call.
Currently safe but fragile. TCP transport uses a separate iov copy via
`kvec_array_init()`.

### P4-MC-05: IPv6-only NIC skipped in FSCTL_QUERY_NETWORK_INTERFACE_INFO

In `fsctl_query_iface_info_ioctl()`, the `ipv6_retry:` logic only emits an
IPv6 entry after successfully getting an IPv4 entry. If `__in_dev_get_rtnl()`
returns NULL (no IPv4 address), the whole NIC is skipped via `continue`,
including the IPv6 address. An IPv6-only NIC is never reported.
